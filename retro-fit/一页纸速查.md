# 一页纸速查 · 大模型治理历史代码

> 配套《历史代码治理实践报告》。目标：不读主报告也能直接上手。

## 四层框架（一句话记忆）

**看懂 → 锁住 → 动手 → 固化**
理解层（画地图）→ 守护层（建清单+快照）→ 改造层（小步+审查+灰度）→ 工作流层（文档驱动+CI）

## 不可破坏 · 4 个检查点（改动前逐条问）

- [ ] **客户/租户分支**：有没有按客户、版本、feature flag 走的特殊分支？
- [ ] **默认值与空值**：默认值、null / 空处理是不是历史约定，不能简单统一？
- [ ] **错误码与 fallback**：错误码、重试、超时、降级逻辑是否被下游依赖？
- [ ] **隐藏依赖字段**：有没有「业务不用、但日志 / 监控 / 运维脚本依赖」的字段？

## 验收标准四要素（让 AI 开写前先写清）

| 要素 | 要回答的问题 |
|---|---|
| **背景** | 为什么要改这段？ |
| **目标** | 改完要达到什么效果？ |
| **限制** | 哪些绝对不能动？ |
| **风险** | 哪些地方可能出问题？ |

## Prompt 模板：先盘点，不要改

```
请只阅读 [指定文件/模块]，不要修改代码。输出一份「不可破坏清单」：
1. 当前模块的入口、主流程、外部依赖；
2. 所有看起来像客户定制 / 历史兼容 / 灰度开关 / 特殊错误码 / fallback 的逻辑；
3. 每条特殊逻辑的代码位置和触发条件；
4. 哪些行为在重构中绝对不能悄悄改变；
5. 你不确定的地方，必须标为「不确定」，不要脑补。
重构只能改变结构，不改变行为；任何行为变更必须单独列出，不得混入重构。
最后输出：保留了哪些特殊逻辑、分别对应哪些测试覆盖。
```

## Prompt 模板：改完做行为审查

```
请只审查这次 diff 的「行为风险」，不要评价代码风格：
有没有删掉特殊分支？有没有改默认值、调用顺序、重试次数、超时、fallback？
逐条列出可能改变线上行为的点，并标注影响的字段 / 错误码 / 日志字段。
```

## 黄金法则

> **老代码里的特殊逻辑，先当文物保护，不当垃圾清理——先标出来、测起来、写下来，再改。**

## 三条铁律

1. 先有测试基线（行为快照）再动手——没基线就没有边界。
2. 想和干分开——先出 md 方案，人工确认，再让 AI 写代码。
3. 高风险路径上灰度 + 开关 + 监控——本地通过 ≠ 线上安全。
