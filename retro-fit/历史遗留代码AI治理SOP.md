# 历史遗留代码 AI 治理 SOP

SOP 是 Standard Operating Procedure 的缩写，中文通常译为“标准作业流程”。本文把一次历史遗留代码 AI 治理试点拆成可执行的步骤、输入、产物和验收标准，帮助团队照着跑、跑完复盘、下次复用。

这份 SOP 不是鼓励直接用大模型重构老系统，而是提供一套更稳妥的工作方式：先理解，再守护，再验证，最后才小步改造。

## 1. 适用场景

适合使用本 SOP 的场景：

- 老模块缺文档，只有少数人知道历史背景。
- 模块里存在客户定制、历史补丁、灰度开关、老版本兼容逻辑。
- 团队想用大模型辅助理解代码、补文档、补测试或做小范围重构。
- 改动风险可控，有测试环境、review 流程和基本回滚能力。
- 目标是先做一个试点模块，而不是一次性重写整个系统。

## 2. 不适用场景

以下场景不建议直接进入 AI 改造阶段：

- 没有明确 owner 的核心生产链路。
- 没有测试环境、灰度能力或回滚方案。
- 涉及资金、计费、权限、合规、安全策略等高风险逻辑，但无法充分验证。
- 影响范围不清楚，也无法找到业务方或历史维护者确认。
- 期望大模型一次性自动重写整个老系统。

这些场景下，大模型可以先用于“只读理解”和“风险盘点”，不要直接生成或修改生产代码。

## 3. 角色分工

一次试点建议至少明确以下角色：

| 角色 | 职责 |
|---|---|
| 试点负责人 | 确定试点模块、控制范围、推进阶段验收 |
| 模块 owner | 确认业务意图、特殊逻辑、历史兼容和风险边界 |
| 开发人员 | 使用大模型生成地图、账本、快照和小步改造方案 |
| Reviewer | 审查行为风险，重点看兼容逻辑、错误码、默认值、副作用 |
| 测试 / 质量人员 | 设计回归用例、行为快照、灰度验证和上线观察项 |

如果团队规模较小，可以一人承担多个角色，但必须显式保留 owner 确认和 review 环节。

## 4. 试点前准备

开始前先准备这些输入：

- 试点模块范围：目录、入口文件、核心接口或关键流程。
- 相关文档：README、架构文档、历史设计文档、运维手册。
- 历史线索：PR、commit、issue、工单、事故复盘、客户配置。
- 运行时证据：日志、监控、接口样例、线上配置、典型请求。
- 现有验证手段：单测、集成测试、回归脚本、手工验收 checklist。

准备阶段的关键原则：不要把整个仓库一次性塞给大模型。先给入口和主流程，再按问题补上下文。

## 5. 第一周怎么启动

第一周的目标不是立刻改代码，而是完成一次可控试点的启动准备：选定模块、明确边界、收集上下文、产出第一版模块地图和风险清单。

### 第 1 天：确定试点模块

选择一个“有代表性但不致命”的老模块作为试点。

优先选择：

- 业务上仍在维护，近期可能有需求改动。
- 有一定历史包袱，比如客户定制、兼容逻辑、异常兜底。
- 影响范围相对可控，有 owner 可以确认。
- 有测试环境或至少有手工验证方式。

暂时避免：

- 无人负责的核心链路。
- 影响资金、权限、计费、安全策略的高风险模块。
- 没有回滚手段、没有测试环境的线上关键路径。
- 一次性覆盖多个系统的大范围改造。

### 第 2 天：明确试点边界

把试点范围写清楚，避免大模型和团队都发散。

需要明确：

- 本次只看哪个目录、接口或流程。
- 本次不看哪些模块。
- 本次目标是理解、补测试、局部重构，还是为后续改造做准备。
- 哪些行为明确不能改变。
- 哪些问题需要 owner 或业务方确认。

建议输出一份 `试点范围说明.md`。

### 第 3 天：收集上下文材料

不要只把代码丢给大模型。第一周要先把上下文收齐。

建议收集：

- 入口文件、核心 service、关键类型定义。
- README、架构文档、历史设计文档。
- 最近相关 PR、commit、issue、工单。
- 典型请求、接口响应、日志样例。
- 线上配置、灰度配置、客户配置。
- 现有测试、回归脚本、手工验收步骤。

如果材料不完整，也要记录“不完整在哪里”。

### 第 4 天：生成第一版模块地图

让大模型只读代码，不修改代码，输出第一版模块地图。

模块地图至少包括：

- 模块入口。
- 主流程调用链。
- 核心数据结构。
- 外部依赖。
- 高风险区域。
- 不确定点。
- 下一步最应该继续阅读的文件。

这一天的重点是看懂，不是评价代码好坏。

### 第 5 天：生成第一版分支账本

基于模块地图，让大模型扫描特殊逻辑，生成第一版分支账本。

重点关注：

- 客户 / 租户判断。
- 老版本兼容。
- feature flag。
- 特殊错误码。
- fallback / retry / timeout。
- 默认值和空值处理。
- 日志、监控、运维脚本依赖字段。
- DB、MQ、HTTP 调用等副作用。

每条记录都要包含代码位置、触发条件、影响行为和不确定点。

### 第一周结束时的验收标准

第一周结束时，不要求已经完成改造，但应该拿到这些产物：

- `试点范围说明.md`
- `模块地图.md`
- `分支账本.md`
- `待确认问题清单.md`
- 初步的 `不可破坏清单`
- 下一周要补的行为快照或回归测试列表

如果这些产物还没有，就不要急着进入代码改造。

### 第一周不要做什么

- 不要一上来让大模型“重构整个模块”。
- 不要在没有模块地图前大范围改代码。
- 不要把不确定的历史逻辑当成可删除逻辑。
- 不要把行为变更混进“整理代码”的 PR。
- 不要只在对话里说“这个别动”，而不沉淀到清单、测试或注释里。

一句话：第一周只做启动和考古，不做大改；目标是把未知变少，把风险显性化。

## 6. 阶段一：生成模块地图

### 目标

让团队先知道模块从哪里进、怎么流转、依赖谁、哪些地方最危险。

### 输入

- 模块入口文件或入口接口。
- 核心 service / handler / command。
- 关键类型定义、配置文件和外部依赖说明。

### 操作

让大模型只读代码，不修改代码，输出模块地图：

- 对外暴露的入口。
- 主要文件、类、函数的职责。
- 主流程调用链。
- 核心数据结构和状态流转。
- 外部依赖：DB、缓存、MQ、HTTP/RPC、配置、第三方服务。
- 高风险区域和不确定点。
- 下一步最应该阅读的文件列表。

### 产物

- `模块地图.md`
- 不确定点清单
- 下一批阅读文件列表

### 验收标准

- 模块 owner 能确认主流程基本正确。
- 每个关键结论都有代码位置或运行时证据。
- 不确定点被明确标出，没有把猜测写成事实。

## 7. 阶段二：建立分支账本和不可破坏清单

### 目标

把隐藏在老代码里的客户定制、历史兼容、异常兜底、灰度开关等特殊逻辑显式化。

### 输入

- 模块地图。
- 主流程相关代码。
- 历史 PR / 工单 / 客户配置 / 事故记录。
- 典型请求、日志和配置样例。

### 操作

让大模型生成分支账本，每条记录至少包括：

- 分支条件。
- 代码位置。
- 触发场景。
- 影响的输出、副作用、错误码、日志、DB、MQ 或外部调用。
- 疑似类型：默认逻辑、客户定制、历史兼容、灰度开关、异常兜底、不确定。
- 如果重构时丢掉它，可能造成什么后果。
- 建议用什么测试或行为快照保护。

然后由模块 owner 和 reviewer 从分支账本中提炼不可破坏清单：

- 必须保持兼容的行为。
- 可以后续清理但本次不动的行为。
- 可以删除或改造的历史逻辑。
- 仍需补充证据的不确定逻辑。

### 产物

- `分支账本.md`
- `不可破坏清单.md`
- 待确认问题列表

### 验收标准

- 客户 / 租户 / 版本 / feature flag / 错误码 / fallback / 空值处理等高风险分支已覆盖。
- 不可破坏清单经过人工确认。
- 不确定项没有被直接当作可删除逻辑处理。

## 8. 阶段三：建立行为快照或回归测试

### 目标

先锁住旧代码当前行为，再让大模型参与改造。

### 输入

- 不可破坏清单。
- 现有测试、接口样例、日志样例。
- 可 mock 的外部依赖说明。

### 操作

为高风险分支设计最小行为快照集。每个 case 应包含：

- 输入条件。
- 需要 mock 的依赖。
- 预期输出。
- 预期副作用：DB 写入、缓存、MQ、HTTP 调用、日志字段。
- 覆盖的分支账本记录。
- 为什么这个 case 重要。

如果暂时无法自动化，至少沉淀为手工验收 checklist 或接口响应快照。

### 产物

- 行为快照用例
- characterization test / 回归测试
- 手工验收 checklist

### 验收标准

- 不可破坏清单中的高风险行为都有测试、快照或手工验收项保护。
- 测试目标是“旧行为一致”，不是重新设计理想行为。
- 无法自动化的验证项有明确负责人和执行方式。

## 9. 阶段四：小步改造

### 目标

在保护网存在的前提下，只做必要范围内的小步改造。

### 输入

- 模块地图。
- 不可破坏清单。
- 行为快照或回归测试。
- 本次明确的改造目标。

### 操作

改造时遵守以下规则：

- 一次只改一个目标，不顺手重构无关代码。
- 重构和行为变更分开提交。
- 不改变 URL、错误码、默认值、空值处理、日志字段、序列化格式等兼容契约。
- 如必须改变行为，单独列为“显式行为变更”。
- 改完后让大模型只做 diff 行为审查，不评价代码风格。

行为审查重点：

- 是否删除或合并了特殊分支。
- 是否改变默认值、错误码、异常处理。
- 是否改变调用顺序、重试、超时、fallback。
- 是否改变 DB、缓存、MQ、HTTP 调用等副作用。
- 是否改变日志字段、回调字段、序列化格式。

### 产物

- 小范围代码改动
- diff 行为审查记录
- 显式行为变更清单
- 更新后的测试 / 快照

### 验收标准

- 本次改动范围和目标一致。
- 不可破坏清单中的行为未被悄悄改变。
- 所有显式行为变更都被单独说明并经过 owner 确认。
- 相关测试、快照或手工验收已执行。

## 10. 阶段五：灰度上线与复盘

### 目标

把安全验证从本地和测试环境延伸到真实运行环境。

### 输入

- 改造 PR。
- 测试结果。
- 灰度方案。
- 回滚方案。
- 监控和日志观察项。

### 操作

上线前确认：

- 是否需要 feature flag 或新旧入口并行。
- 是否可以小流量灰度。
- 是否有关键指标：错误率、超时率、业务转化、核心日志、下游回调成功率。
- 是否有明确回滚方式。
- 是否有人在灰度窗口内观察。

上线后复盘：

- 哪些行为快照有效发现了风险。
- 哪些特殊逻辑是在过程中才发现的。
- 哪些文档、测试或清单需要更新。
- 下一次试点可以复用哪些模板。

### 产物

- 灰度计划
- 回滚计划
- 上线观察记录
- 试点复盘记录

### 验收标准

- 灰度期间关键指标无异常。
- 出现异常时可以快速回滚。
- 模块地图、分支账本、不可破坏清单和测试资产已回流仓库。

## 11. 最终交付物清单

一次完整试点建议至少交付：

- 模块地图
- 分支账本
- 不可破坏清单
- 行为快照 / 回归测试 / 手工验收 checklist
- 小步改造 PR
- diff 行为审查记录
- 显式行为变更清单
- 灰度和回滚方案
- 试点复盘记录

## 12. 总体验收标准

试点完成时，应满足：

- 团队能说明该模块的入口、主流程、外部依赖和高风险区域。
- 高风险特殊逻辑已被标记、确认并有保护方式。
- 行为变更和结构重构没有混在一起。
- 大模型产物不只停留在对话里，而是沉淀到文档、测试、PR 或代码注释中。
- 后续再改同一模块时，不需要从零开始考古。

## 13. 与配套材料的关系

- `历史代码治理实践报告.md`：解释为什么需要这套方法，以及背后的数据、案例和治理框架。
- `一页纸速查.md`：提供快速 checklist 和 Prompt 模板。
- 本文：把一次试点拆成可执行流程，说明每一步的输入、动作、产物和验收标准。
